# 项目版本与迭代流程

## 当前基线

- 正式版基线：`1.0.1`
- GitHub 正式版标签：`v1.0.1`
- 本地备份目录：`backups/v1.0.1/`

## 版本命名规则

从 `1.0.1` 正式版继续迭代时，使用以下格式：

- `1.0.1-r1`
- `1.0.1-r2`
- `1.0.1-r3`

含义：
- `1.0.1` = 当前确认过的正式基线版本
- `-rN` = 基于该正式版的第 N 次迭代修订

如果未来做出一轮足够稳定、可单独发布的更新，再升级为新的正式版本，例如：
- `1.0.2`
- 然后继续 `1.0.2-r1`

## 每次更新的固定流程（必须执行）

1. **先备份当前版本**
   - 本地复制到：`backups/v当前版本/`
   - 至少保存：`index.html`、`README.md`
   - 写入 `manifest.json` 记录来源 commit 和时间
   - 可直接运行：`./scripts/backup_version.sh <版本号> "备注"`

2. **GitHub 保存原版本**
   - 为备份前的状态打 tag，例如：`v1.0.1`、`v1.0.2`
   - 如只是迭代修订，可继续使用已有正式版 tag + main 分支提交历史 + 本地 backups 目录组合保存

3. **再开始修改**
   - 修改页面中的 `APP_VERSION`
   - 保持页面右上角/不显眼位置显示版本号

4. **本地测试通过后才能提交**
   - 至少做 smoke test：
     - 打开页面
     - 点击“+ 添加项目”
     - 创建项目
     - 编辑项目
     - 删除测试项目
     - 检查导出/导入入口是否还在
     - 检查列表 / 看板 / 日历切换

5. **提交并推送到 GitHub**
   - commit message 需要写清楚修复或功能内容
   - push 前确保工作区干净、测试完成
   - 可直接运行：`./scripts/release.sh <下一个版本号> "commit message" [--push] [--yes]`

## Codex / GitHub 改动如何并入这套系统

### 推荐方式（首选）

1. **Codex 在分支或 PR 上工作，不直接改 main**
2. 把 Codex 的改动先合并到本地仓库
3. 在本地执行本流程：
   - 先备份当前 main 对应版本
   - 本地测试
   - 确认版本号
   - 再 merge / push

### 如果 Codex 已经直接推到了 GitHub

按下面方式补救并纳入流程：

1. 先 `git pull` 同步本地
2. 判断该提交属于：
   - 正式版更新，还是
   - 迭代修订
3. 立即补建：
   - 本地 `backups/v.../`
   - 对应 Git tag（如果缺失）
4. 本地跑 smoke test
5. 测试通过后，再继续后续迭代

## 原则

- **不跳过本地备份**
- **不跳过本地测试**
- **不直接把未经验证的改动当正式版**
- **GitHub 是远端留档；本地 backups 是本地快照；二者都保留**
